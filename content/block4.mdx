## 4-1. 데모 (30분)

**방식**: 자원자 3~4명, 각 5~7분

<Steps>
### 발표 틀

아래 4가지를 중심으로 공유하세요:

```
1. 어떤 카드를 골랐나? (또는 어떤 문제를 가져왔나?)
2. 오늘 만든 것 (CLAUDE.md, 프롬프트 템플릿, 구현 결과 등)
3. 가장 놀랐던 순간은?
4. 가장 실패했던 시도는?
```

### 청중 관점에서 볼 것

발표를 들으며 아래를 생각해보세요:

- 저 접근 방식을 내 업무에 쓸 수 있나?
- 내 CLAUDE.md에 추가할 만한 것이 있나?
- 실패 사례에서 나도 같은 실수를 하고 있진 않나?
</Steps>

## 4-2. 개인 회고 (15분)

조용히 혼자 작성합니다.

```
[오늘의 회고]

1. 오전에 뽑은 불편함 중, 오늘 건드린 것은?


2. 오늘 배운 것 중 내일 당장 쓸 수 있는 것 1가지:


3. 다음 1주 안에 할 것:
   - 언제:
   - 무엇을:
   - 어떻게 확인할지:


4. CLAUDE.md에 오늘 추가한 항목:
```

<Callout type="warning">
**작성 팁**: "언제"를 구체적으로 쓸수록 실행 가능성이 높아집니다.
"조만간"이 아니라 "이번 주 수요일 PR 작업 시"처럼 구체적으로 적어보세요.
</Callout>

## 4-3. 마무리 (15분)

### 오늘의 핵심 3가지

**1. Context가 먼저다**

> CLAUDE.md 없이 에이전트에게 코드를 시키는 것은 신입에게 문서 없이 일을 시키는 것

- AI는 맥락을 알 때 비로소 제대로 일한다
- CLAUDE.md는 한 번 만들면 매번 효과를 낸다 — **복리 효과**

**2. 계획이 실행을 결정한다**

> *"A good plan handed to Claude is basically free — a bad plan wastes hours."*
>
> — Boris Cherny, [How Boris Uses Claude Code](https://howborisusesclaudecode.com) (2026)

- 복잡한 작업에서 Plan Mode는 선택이 아니라 기본값
- 계획을 검토하는 2분이 디버깅 2시간을 막는다

**3. 검증 루프가 품질을 만든다**

> *"Give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result."*
>
> — Boris Cherny, [How Boris Uses Claude Code](https://howborisusesclaudecode.com) (2026)

- AI에게 자기 결과를 확인할 방법을 주면 품질이 2~3배 올라간다
- 테스트가 있는 코드베이스에서 AI도 더 잘 일한다
