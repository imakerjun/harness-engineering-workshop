# 카드 C: 새 기능을 안전하게 추가하기 ★★☆

> **대상**: 기능 구현이 주 업무이고 AI를 더 안전하게 쓰고 싶은 개발자

## 📖 시나리오

> 기존 코드에 새 기능을 추가해야 합니다.
> Plan Mode + Verification Loop로 기존 동작을 깨지 않고 안전하게 추가합니다.

<Callout type="info" emoji="🎯">
**어떤 하네스 패턴을 쓰는가**

- **Context**: `CLAUDE.md` / `AGENTS.md` 로 기존 컨벤션 유지
- **Plan**: Plan Mode (또는 명시적 plan 프롬프트)로 범위와 순서 먼저 확정
- **Verify**: 기존 테스트 + 새 테스트 모두 통과 확인
</Callout>

<Callout emoji="🛤️">
**두 가지 트랙 — 본인 수준에 맞게 선택**

| 트랙 | 대상 코드 | 추천 대상 |
|---|---|---|
| **기본 트랙** | `sds-harness-lab` 에 아래 요구사항 카드 중 하나 추가 | 처음 시도. Plan → Verify 사이클을 작은 코드에서 끝까지 돌리기 |
| **도전 트랙** | **Aider** 또는 **Codex CLI** 의 한 모듈에 작은 개선 추가 (오타·작은 메시지 보강 수준) | 실전 규모에서 Plan/Verify가 어떻게 다른지 직접 보고 싶을 때 |

도전 트랙은 "큰 기능 추가"가 아닙니다 — 한 줄~수십 줄 정도의 작은 보강에서 **Plan과 Verify가 어떻게 다르게 작동하는지** 관찰하는 게 목표입니다.
</Callout>

### 도전 트랙 — 분석/기여 대상 선택

<Tabs items={['Aider (Python)', 'Codex CLI (TypeScript)']}>
  <Tabs.Tab>
    **[paul-gauthier/aider](https://github.com/paul-gauthier/aider)** 에 작은 개선을 시도해보세요.

    예시 시도:
    - 특정 에러 메시지를 더 명확하게 (예: `aider/commands.py` 의 사용자 친화 메시지)
    - `--help` 출력에 짧은 예시 추가
    - 한국어 로케일 문서 한 줄 추가

    **Plan 프롬프트 예시**:
    ```
    aider/commands.py 의 /add 명령에서 존재하지 않는 파일을 추가하려 할 때
    나오는 에러 메시지를 더 명확하게 바꾸려고 해. 변경하지 말고 계획만 먼저 보여줘 —
    어떤 파일을 어디서 어떻게 수정할지, 그리고 기존 테스트 중 어떤 것이 영향을 받을지.
    ```
  </Tabs.Tab>
  <Tabs.Tab>
    **[openai/codex](https://github.com/openai/codex)** 에 작은 개선을 시도해보세요.

    예시 시도:
    - 슬래시 명령 도움말 정렬·문구 다듬기
    - 특정 에러 경로의 로그 메시지 보강
    - README 작은 보강

    **Plan 프롬프트 예시**:
    ```
    Codex의 /help 명령 출력에 짧은 예시 1줄을 명령마다 추가하려고 해.
    변경하지 말고 계획만 먼저 보여줘 — 어떤 파일에 어떤 형식으로 추가할지,
    기존 테스트·스냅샷이 어떻게 영향받는지.
    ```
  </Tabs.Tab>
</Tabs>

<Callout type="warning" emoji="⚠️">
**도전 트랙 주의 — 실제 PR이 목적이 아님**

큰 오픈소스에 PR을 만드는 게 목표가 아닙니다. 본인 fork에서 작은 변경을 직접 시도하면서 **Plan → Verify 사이클이 큰 코드에서 어떻게 다르게 느껴지는지** 체험하는 게 카드 C 도전 트랙의 핵심입니다.
</Callout>

## 📋 요구사항 카드

아래 중 하나를 선택하거나, 블록 1에서 뽑은 본인의 문제를 사용하세요.

**옵션 1 — 페이지네이션 추가**
```
GET /projects 엔드포인트에 페이지네이션을 추가하세요.
- query parameter: page (기본값 1), size (기본값 10)
- 응답에 total, page, size, items 포함
- page나 size가 0 이하이면 400 반환
```

**옵션 2 — 상태 전이 검증**
```
프로젝트 상태 변경에 규칙을 추가하세요.
- todo → in_progress → done 순서만 허용
- archived는 any → archived 가능
- 규칙 위반 시 400 반환 + 현재 상태와 시도한 상태 메시지 포함
```

**옵션 3 — 검색 기능 추가**
```
GET /projects에 검색 기능을 추가하세요.
- query parameter: q (이름 또는 설명에 포함된 문자열)
- query parameter: status (해당 상태만 필터)
- 두 조건 동시 사용 가능
```

## 1️⃣ Phase 1: 요구사항 분석 (30분)

<Steps>
### 직접 먼저 분석

구현 전에 직접 생각해보세요:
- 이 기능에서 예상되는 엣지 케이스는?
- 건드리면 안 되는 기존 동작은?
- 완료 기준은 무엇인가?

### AI에게 분석 요청

```
아래 요구사항을 구현하기 전에,
예상되는 엣지 케이스와 기존 코드에서 영향받을 부분을 분석해줘.

{선택한 요구사항 붙여넣기}
```

비교: 내가 생각한 것 vs AI가 발견한 것
</Steps>

## 2️⃣ Phase 2: Plan Mode로 설계 (45분)

<Steps>
### Plan Mode 진입

<Tabs items={['Claude Code', 'Codex', 'AI Pro']}>
  <Tabs.Tab>
    `Shift + Tab` → Plan Mode

    ```
    다음 요구사항을 구현하기 위한 계획을 만들어줘:
    {선택한 요구사항}
    ```
  </Tabs.Tab>
  <Tabs.Tab>
    명시적 프롬프트로 plan을 먼저 받습니다:
    ```
    변경하지 말고 계획만 먼저 보여줘.
    요구사항: {선택한 요구사항}
    포함할 것: 변경할 파일·순서·완료 검증 방법
    ```
  </Tabs.Tab>
  <Tabs.Tab>
    계획 검토 기능 또는 명시적 프롬프트:
    ```
    먼저 계획만 보여줘. 코드는 쓰지 마.
    요구사항: {선택한 요구사항}
    ```
  </Tabs.Tab>
</Tabs>

### 계획 검토

계획이 나오면 실행 전에 반드시 확인:

- [ ] 변경할 파일 목록이 명시됐나?
- [ ] 기존 테스트를 깨지 않는다고 명시했나?
- [ ] 완료 확인 방법(테스트 실행)이 포함됐나?
- [ ] 내가 분석한 엣지 케이스가 계획에 반영됐나?

빠진 것이 있으면 수정 요청:

```
계획에 엣지 케이스 처리({엣지 케이스 내용})를 추가해줘.
그리고 완료 후 기존 테스트가 모두 통과하는지 확인하는 단계도 포함해줘.
```

### 승인 후 실행
</Steps>

## 3️⃣ Phase 3: 구현 + 검증 (60분)

<Steps>
### 실행 후 검증 요청

```
구현이 끝나면 pytest를 실행하고,
모든 테스트가 통과할 때까지 수정해줘.
기존 테스트가 깨지면 반드시 알려줘.
새 기능에 대한 테스트도 함께 작성해줘.
```

### 엣지 케이스 테스트

```
{선택한 기능}에서 발생할 수 있는 경계 케이스를
직접 테스트하고 결과를 알려줘.

예: page=0, size=-1, 존재하지 않는 상태값 등
```

</Steps>
