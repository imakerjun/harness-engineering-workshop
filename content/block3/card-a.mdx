- **Verify**: 테스트를 추가하면서 AI가 직접 실행하고 검증

## 1️⃣ Phase 1: 탐색과 컨텍스트 파일 작성 (60분)

이 Phase의 산출물은 **README가 아니라 `CLAUDE.md` / `AGENTS.md`** 입니다. 이유: 카드 A의 학습 목표는 "다음 세션의 AI가 빠르게 시작할 수 있는 컨텍스트를 만드는 것" — 즉 사람용 문서(README)가 아니라 **AI용 컨텍스트 파일**입니다.

<Callout type="info" emoji="📚">
**출처와 근거**

- Anthropic Claude Code 공식 가이드 — `/init`은 README가 아니라 `CLAUDE.md` 초안을 생성합니다.
- Birgitta Böckeler, [Harness engineering for coding agent users](https://martinfowler.com/articles/harness-engineering.html) — 하네스의 첫 축은 "the context it receives".
- Boris Cherny, [How Boris Uses Claude Code](https://howborisusesclaudecode.com) — CLAUDE.md는 git에 체크인되는 팀 자산. 실수를 누적해 같은 실수가 반복되지 않게.
- 강규한, [AI와 함께하는 테스트 자동화](https://techblog.woowahan.com/24568/) — 검증 부재 = 컴파일 성공률 10%. 검증 가능한 사실만 기록해야 하는 이유.
</Callout>

<Steps>
### Step 1 — 큰 그림만 (15분, Onion Layer 1)

전체 파악을 한 번에 시도하지 않습니다. **디렉터리 단위의 큰 그림**만 먼저 받습니다.

```
이 프로젝트의 디렉터리 구조를 보여주고, 각 최상위 폴더가
어떤 역할을 하는지 한 줄로만 설명해줘.

규칙:
- 확실하지 않은 추측에는 [?] 표시
- 각 설명 끝에 근거가 되는 파일 경로 1개 표기 (예: 출처: app/main.py)
- 200자 이내로 답해줘
```

**왜 이렇게**: Aider의 repo map 패턴 — 큰 코드일수록 첫 패스는 "전체 한 줄 요약"만. 자세한 건 작업 시점에 just-in-time으로 가져옵니다 ([Birgitta Böckeler](https://martinfowler.com/articles/harness-engineering.html)).

### Step 2 — 핵심 모듈 1개 깊이 (20분, Onion Layer 2)

큰 그림에서 가장 자주 변할 것 같은 모듈 1개를 골라 진입점 흐름을 추적합니다.

```
{선택한 모듈, 예: app/services/}의 진입점 함수가 무엇이고,
사용자 요청이 들어왔을 때 어떤 호출 흐름으로 흘러가는지 설명해줘.

규칙:
- 함수명·파일 경로·라인 번호 명시
- 확실하지 않은 부분은 [?]로 표시
- 본인이 직접 읽은 코드만 인용해 — 추측한 함수명은 절대 만들지 마
```

읽고 나서 학습자가 직접 검증 (Step 3).

### Step 3 — 검증 가능한 사실만 기록 (15분)

처음 보는 큰 코드에서 "AI 설명이 다 맞는지"는 학습자도 알 수 없습니다. 대신 **기계적으로 검증 가능한 것**만 골라서 확인합니다.

```bash
# AI가 언급한 파일이 실제로 있는가
ls path/to/file.py

# AI가 언급한 함수가 실제로 정의돼 있는가
grep -rn "def function_name" path/

# 동작 주장이 있다면 테스트가 보장하는가
pytest tests/ -v -k "function_name"
```

검증 결과를 분류:
- ✅ **확인된 사실** — 컨텍스트 파일에 그대로 기록
- ❓ **확인 불가** — 컨텍스트 파일에 `[?]` 또는 "TBD"로 보류
- ❌ **틀린 추측** — 즉시 "AI가 자주 헤매는 것" 섹션에 추가 (Compounding)

### Step 4 — CLAUDE.md / AGENTS.md 초안 작성 (10분)

위 3단계 결과를 컨텍스트 파일에 정리합니다. 200줄 한도를 의식하며.

최소 포함 항목:
- [ ] 이 프로젝트가 뭐 하는 건지 (1줄)
- [ ] 디렉터리 최상위 한 줄 설명 (Step 1 결과)
- [ ] 핵심 모듈 진입점 함수와 호출 흐름 (Step 2 결과)
- [ ] **AI가 자주 헤매는 것** — 빈 섹션이라도 만들어두기 (Compounding 출발점)
- [ ] **건드리지 말 것** — Step 3에서 발견한 것 또는 [?] 영역
</Steps>

<Callout emoji="🛤️">
**도전 트랙 — Sub-agent로 컨텍스트 격리하세요**

큰 규모에서 한 번에 전체를 보려는 시도가 **AI가 헤매기 시작하는 가장 흔한 지점**입니다. HumanLayer의 ["Skill Issue: Harness Engineering for Coding Agents"](https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents) 에서 권장하는 패턴 — **Sub-agent를 context firewall로 쓰기**:

> *"Sub-agents function as a context firewall that ensures discrete tasks can run in isolated context windows so none of the intermediate noise accumulates in your parent thread."*

도전 트랙 학습자는 위 Step 1~4를 다음과 같이 변형합니다:

1. **Step 1 변형 — Sub-agent에게 위임**
   - "이 코드베이스의 디렉터리 구조와 각 최상위 폴더 역할을 한 줄씩 요약해서 가져와줘. 너의 탐색 과정은 보여주지 말고 결과 요약만." (메인 컨텍스트에 노이즈가 안 쌓임)
2. **Step 2 변형 — 모듈 1개만 골라서 깊이**
   - 위 추천 진입점(`aider/coders/base_coder.py` 또는 `src/`) 중 한 개만. 전체 코드베이스를 다 읽지 않음.
3. **Step 3 변형 — 검증을 더 까다롭게**
   - 큰 코드에서는 AI 환각 빈도가 더 높습니다. AI가 인용한 모든 함수명·파일 경로를 한 번씩 grep 검증.
4. **Step 4 변형 — 한계를 운영 노트로**
   - "AI가 헤매기 시작한 지점" 을 별도 섹션으로. 다음 세션에서 그 영역은 사람이 직접 읽고 답합니다.

**관찰 노트** (블록 4 발표 재료):
- 가장 도움이 됐던 프롬프트 한 줄:
- AI가 가장 자주 헤맸던 영역:
- 컨텍스트 한계가 처음 느껴진 시점 (어떤 파일/몇 번째 질문):
- Sub-agent 위임이 메인 컨텍스트 보존에 얼마나 효과적이었나:
</Callout>

## 2️⃣ Phase 2: 테스트 추가 (90분)

<Steps>
### 테스트 없는 함수 목록 파악

```
이 프로젝트에서 테스트가 없는 핵심 함수를 찾아서
우선순위 순으로 목록을 알려줘.
```

### 함수별 테스트 작성 요청

각 함수마다 아래 패턴을 반복합니다:

```
{함수명}에 대한 단위 테스트를 작성해줘.

테스트 작성 전에 이 함수가 무엇을 해야 하는지 먼저 설명해줘.
작성 후 pytest로 통과 여부를 직접 확인해줘.
실패하면 코드를 수정해서 통과시켜줘.
```

<Callout type="warning">
**주의**: AI가 None 처리, 빈 입력, 없는 ID 조회 같은 엣지 케이스를 빠뜨리는 경우가 많습니다.
테스트 목록을 보고 "이런 경우는?" 하고 추가 요청하세요.
</Callout>

### AI가 발견한 버그 기록

AI가 테스트를 작성하다가 버그를 발견하면, CLAUDE.md에 기록합니다:

```markdown
## ⚠️ 자주 하는 실수
- 없는 ID 조회 시 None.to_dict() 호출 → AttributeError 발생
  → 반드시 None 체크 후 404 반환
```
</Steps>
