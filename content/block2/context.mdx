# 2-1: Context Engineering

> **10:00~11:00 | 강의 20분 + 실습 40분**

## 왜 컨텍스트인가

> *"Context engineering is the art of providing the right information, in the right format, at the right time, so the model can do useful work."*
>
> — Andrej Karpathy, [X (Twitter)](https://x.com/karpathy) (2025)

AI가 엉뚱한 코드를 생성하는 이유:

1. **팀 컨벤션을 모른다** — Kotest? JUnit? given-when-then? 알려주지 않으면 추측한다
2. **코드베이스 경계를 모른다** — 어떤 클래스가 실제로 있는지, 어떤 패턴을 쓰는지 모른다
3. **완료 기준을 모른다** — 사람이 정의하지 않으면 AI가 스스로 낮은 기준을 만든다

해결책: **CLAUDE.md** — AI에게 주는 팀 설명서.

> Boris Cherny(Claude Code 창시자):
> *"Each team at Anthropic maintains a CLAUDE.md in git to document mistakes, so Claude can improve over time, and best practices, such as style conventions, design guidelines, PR template."*
>
> 출처: [How Boris Uses Claude Code](https://howborisusesclaudecode.com) (2026)

## 컨텍스트의 3가지 형태

CLAUDE.md 하나만이 컨텍스트가 아니다. 3가지 형태로 AI에게 맥락을 줄 수 있다.

| 형태 | 도구 | 역할 |
|---|---|---|
| **규칙 문서** | `CLAUDE.md` | 항상 적용되는 팀 컨벤션, 금지 사항 |
| **커스텀 명령어 (Skills)** | `.claude/commands/*.md` | 반복 작업을 슬래시 명령으로 재사용 |
| **MCP 서버** | `npm install` | 팀 전체 컨텍스트를 패키지로 배포 |

오늘 실습에서는 **CLAUDE.md**를 만든다. Skills와 MCP는 CLAUDE.md가 팀 수준으로 확장된 형태다 — [블록 3 카드 B](/block3/card-b)에서 이어진다.

### Skills — 반복 프롬프트를 명령어로

```
# .claude/commands/review.md
# /review 라고 입력하면 이 프롬프트가 실행된다

이 코드의 변경사항을 리뷰해줘.
체크 항목:
- 테스트 커버리지가 충분한가?
- 팀 컨벤션(CLAUDE.md)을 지켰는가?
- 예외 처리가 명확한가?
```

파일을 만들면 Claude Code에서 `/review`로 바로 실행된다. **반복하는 프롬프트는 명령어로 만들어라.**

## 컨텍스트 계층

| 계층 | 위치 | 용도 |
|---|---|---|
| **Enterprise** | OS 시스템 경로 | 회사/IT 정책, override 불가 |
| **전역** | `~/.claude/CLAUDE.md` | 내 개인 규칙 (모든 프로젝트 적용) |
| **프로젝트** | `./CLAUDE.md` | 팀 컨벤션 (git에 체크인) |
| **프로젝트 개인** | `./CLAUDE.local.md` | 나만 쓰는 메모 (git 제외) |
| **세션** | 대화 중 입력 | 일회성 지시 |

충돌 시 우선순위: Enterprise → 프로젝트 → 전역. 예를 들어 회사가 "raw SQL 금지"를 Enterprise 정책으로 설정하면 어떤 프로젝트 CLAUDE.md도 이를 override할 수 없습니다.

## Pre-load vs Just-in-Time

컨텍스트를 언제 주느냐가 품질을 결정합니다.

| 전략 | 무엇인가 | 언제 |
|---|---|---|
| **Pre-load** | 세션 시작 시 미리 주입 | 모든 세션에 공통 필요한 규칙·컨벤션 |
| **Just-in-Time** | 작업 중 필요할 때 로드 | 작업마다 다른 파일, 양이 큰 정보 |

Claude Code의 실제 동작 방식:
- `CLAUDE.md` → **Pre-load** (세션 시작 시 전체 로드)
- 코드 파일 → **Just-in-Time** (`Read`/`Grep`으로 작업 중 로드)

**모든 걸 Pre-load하면 토큰 예산이 빠르게 소모**됩니다. 반대로 모든 걸 JIT로 하면 매번 해맵니다. CLAUDE.md는 "항상 필요한 것만" 담고, 나머지는 AI가 찾아오게 합니다.

## CLAUDE.md 핵심 구성

좋은 CLAUDE.md는 **WHY → WHAT → HOW** 순서로 시작합니다. 컨벤션 나열부터 시작하면 AI가 의도를 추정하다 엉뚱한 방향으로 갑니다.

**흔한 첫 실수:**
- ❌ "완벽하게 써보자" → 한 달째 빈 파일
- ❌ 모든 컨벤션을 한 번에 → 너무 길어서 아무도 안 읽음
- ✅ **20분 안에 1버전을 만들고, 계속 고쳐나가는 것이 정답**

```markdown
# CLAUDE.md

## 이 레포가 뭐 하는 건지 (1~2줄)
프로젝트 관리 REST API. FastAPI + SQLite.

## 기술 스택 & 주요 명령어
- Python 3.11 / FastAPI / pytest
- 테스트: `python3 -m pytest tests/ -v`
- 서버: `uvicorn app.main:app --reload`

## 코딩 컨벤션 (지금 당장 지켜야 할 것 5개)
- 함수명: snake_case
- 반환 형식: dict (to_dict() 메서드 활용)
- 에러: HTTPException 사용 (직접 raise 금지)
- status 허용값: todo, in_progress, done, archived
- 테스트 없이 PR 생성 금지

## AI가 자주 헤매는 것 ⭐ 가장 가치 있는 섹션
- (실습 중 발견할 때마다 추가)
- 예: "status 필드에 다른 값 입력하려 함 → 위 컨벤션 참고"

## 건드리면 안 되는 것
- legacy/ 폴더 (리팩터링 금지)
- .env 파일 (읽기/쓰기 금지)
```

<Callout type="info">
**Compounding Engineering**: Boris Cherny의 핵심 원칙.
Claude가 실수할 때마다 "AI가 자주 헤매는 것" 섹션에 추가하면, 같은 실수가 반복되지 않는다.
이 섹션은 팀 컨벤션 문서에는 없다 — **CLAUDE.md에서만 쌓인다.**
팀 전체가 AI 실수를 공유하고 누적하면 팀 전체의 AI 품질이 올라간다.

**운영 원칙**: AI가 또 같은 실수를 하면 → 그 자리에서 규칙 추가. 월 1회 낡은 규칙 정리. **200줄을 넘기지 말 것** — 길어지면 중요 규칙이 묻힌다.
</Callout>
