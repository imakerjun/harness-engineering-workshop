## 이 교육이 끝나면

교육 종료 시점에 참가자는 아래 상태가 됩니다.

**갖게 되는 것**
- 팀 레포에 커밋할 수 있는 `CLAUDE.md` 초안
- 내일 업무에 바로 쓸 수 있는 프롬프트 패턴 1~3개
- AI가 실수했을 때 어떻게 고치는지에 대한 루틴

**말할 수 있게 되는 것**
- "AI 에이전트가 왜 엉뚱한 결과를 냈는지" 원인을 설명할 수 있다
- "다음 PR에서는 이렇게 쓰겠다"는 구체적인 계획이 있다

**달라지는 것**
- AI를 쓸 때 막연히 결과를 기다리지 않고, 맥락·계획·검증 순서로 접근한다

<Callout type="info">
**핵심 메시지**: AI 에이전트의 성능을 결정하는 것은 모델이 아니라 그것을 감싸는 환경 — **하네스(Harness)** 다.

```
Agent = Model + Harness
```

> *"The performance of an agent is determined not just by the model, but by the harness around it."*
> — Birgitta Böckeler, [Harness engineering for coding agent users](https://martinfowler.com/articles/harness-engineering.html), Thoughtworks (2026.02)
</Callout>

## 왜 하네스인가

AI에게 전부 맡겼는데 뒷수습이 더 오래 걸린 경험, 한 번쯤 있을 것이다.

원인은 3가지다:

1. **AI는 팀 컨벤션을 모른다** — 알려주지 않으면 추측한다
2. **AI는 코드베이스 경계를 모른다** — 어떤 클래스가 실제로 있는지 모른다
3. **AI는 "완료"의 기준을 모른다** — 사람이 정의하지 않으면 스스로 허술한 기준을 만든다

이 3가지는 모두 **하네스가 없어서** 생기는 문제다.

> 우아한형제들 실제 사례: 백엔드 팀이 AI에게 10개 클래스 테스트 생성을 맡겼을 때
> **컴파일 성공률: 10%**. AI의 문제가 아니라 하네스의 부재.
>
> — [AI와 함께하는 테스트 자동화: 플러그인 개발기](https://techblog.woowahan.com/24568/), 강규한 (2025.12)

오늘 배우는 3가지 하네스 패턴:

| 축 | 핵심 질문 | 세션 |
|---|---|---|
| **Context** | AI에게 어떤 맥락을 줄 것인가 | 블록 2-1 |
| **Plan** | AI가 어떻게 실행할 것인가를 먼저 정의하는가 | 블록 2-2 |
| **Verify** | AI가 자기 결과를 확인하는 루프가 있는가 | 블록 2-3 |
